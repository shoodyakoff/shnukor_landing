# MVP: маппинг интерфейса в API каталога

Дата: 2026-05-17

## 1. Общая логика API

Endpoint:

```text
POST https://shnurok-shipping.ru/api/sneakers.php
```

Базовый формат запроса:

```json
{
  "size": ["39.5"],
  "color": ["черный", "серый"],
  "categories": [493],
  "price_from": 10000,
  "price_to": 15000,
  "limit": 20,
  "offset": 0
}
```

Что сейчас понятно из документации и проверок:

- внутри одного фильтра работает `OR`;
- между разными фильтрами работает `AND`;
- `size: ["39", "39.5"]` значит размер 39 ИЛИ 39.5;
- `color: ["красный", "бордовый"]` значит красный ИЛИ бордовый;
- `categories: [493, 494]` значит категория 493 ИЛИ 494;
- категории передаются как ID разделов Bitrix;
- категории ищут товары внутри указанных разделов и их подразделов;
- товар возвращается один раз, даже если подошло несколько SKU;
- если у товара несколько подходящих SKU, API возвращает минимальную подходящую цену и один размер;
- `limit` и `offset` работают;
- пустая выдача приходит как `success: true`, `count: 0`, `items: []`.

Пример текущей логики:

```json
{
  "size": ["39.5"],
  "color": ["черный"],
  "categories": [349, 731, 730, 732, 579],
  "price_from": 10000,
  "price_to": 15000,
  "limit": 20,
  "offset": 0
}
```

Сейчас это читается как:

```text
размер 39.5
И цвет черный
И цена 10 000 - 15 000
И любая из категорий: 349 ИЛИ 731 ИЛИ 730 ИЛИ 732 ИЛИ 579
```

То есть если пользователь лайкнул 5 tinder-карточек, API получает 5 категорий через OR. По ответу разработки от 2026-05-29, при передаче общего `limit` пары теперь выдаются пропорционально по переданным категориям.

Открытые вопросы:

1. Подходит ли для tinder-карточек текущая логика `categories` через OR, или нужна логика пересечения/уточнения?

## 2. Цвета

Все значения ниже проверены через API: метод принимает их и возвращает `count > 0`. Проверка делалась без категории, по всему каталогу, чтобы подтвердить валидность названия цвета. Нижний регистр работает.

## 4. Ценовые категории

Финальный список для UI:

| Категория в UI | Короткое описание | `price_from` | `price_to` |
|---|---|---:|---:|
| До 10 000 ₽ | Базовые модели: доступный вход, простые пары на каждый день или спорт | `0` | `10000` |
| 10 000 - 15 000 ₽ | Повседневные: оптимальный сегмент, больше универсальных и узнаваемых моделей | `10000` | `15000` |

## 5. Сценарии: `На каждый день` и `Для спорта`

Поля `На каждый день` и `Для спорта` сами по себе в API не существуют. Они нужны фронту, чтобы выбрать следующий сценарий и набор tinder-карточек.

В API уходит итоговый массив `categories`, но из-за OR-логики важно заранее понять, какие именно категории можно смешивать в одном массиве.

### 5.1. Если выбран сценарий `На каждый день`

`На каждый день` в MVP должен быть не одной категорией, а списком категорий, которые команда считает повседневными. Нужно получить ID этих категорий для API.

Фронт показывает tinder-карточки из вкладки `На каждый день`. Эти карточки тоже должны иметь свои ID категорий для API.

Открытые вопросы:

1. Какие конкретные категории и ID под капотом считаются сценарием `На каждый день`?
2. Если пользователь выбрал `На каждый день`, но не выбрал ни одну tinder-карточку, передаем все категории, которые считаем дефолтными для `На каждый день`?
3. Если пользователь выбрал хотя бы одну tinder-карточку, что отправляем в API: только ID категорий выбранных карточек или ID выбранных карточек плюс категории, которые считаем `На каждый день`?
4. Могут ли `Массивные` или другие tinder-карточки быть вложены в разные верхние категории, например и в `На каждый день`, и в конкретный вид спорта? Если да, нужно понять, передаем ли верхнюю категорию вместе с категорией карточки и как это выразить в API.
5. Нужен четкий список: название tinder-карточки внутри `На каждый день` -> ID категории или категорий для API.

### 5.2. Если выбран сценарий `Для спорта`

Для спорта нет отдельной категории, которую отправляем всегда. Пользователь выбирает конкретный вид спорта, и в запрос должен уходить ID категории выбранного спорта.

После выбора спорта показываются tinder-карточки, привязанные к этому спорту. По желаемой логике, API должен учитывать и выбранный спорт, и выбранные tinder-карточки. Но в текущем API все значения внутри `categories` работают как OR, поэтому передача `спорт + карточка` в одном массиве может не сузить выдачу, а расширить ее.

| Спорт в UI | Категория сайта | `categories` |
|---|---|---|
| Бег | Беговые кроссовки | `[493]` |
| Фитнес / зал | Фитнесс и кроссфит | `[481]` |
| Футбол | Футбольные бутсы | `[495]` |
| Баскетбол | Баскетбольные кроссовки | `[341]` |
| Теннис | Теннисные кроссовки | `[344]` |
| Трейл / улица / Аутдор | Обувь для хайкинга и трекинга | `[497]` |
| Скейтбординг | Обувь для скейтбординга | `[343]` |
| Универсальные для спорта | Для спорта | `[492]` |

## 10. Противоречия и риски

1. Желаемая логика `спорт + tinder-карточка` выглядит как уточнение, но текущий `categories` работает как OR. Если отправить спорт и карточку в одном массиве, выдача может расшириться, а не сузиться.
2. Та же проблема есть для `На каждый день + tinder-карточка`, если нужно одновременно ограничить выдачу daily-набором и уточнить ее карточкой.
3. Категории из `/catalog/podborki/` могут вернуть не только обувь. Их нельзя считать безопасными, пока не подтверждено обратное или пока API не умеет ограничивать их верхней категорией.
4. Если пользователь лайкнул все tinder-карточки, текущий API вернет общий список по OR. Нужно понять, устраивает ли бизнес такая широкая выдача.
5. Технический список размеров подтвержден API, но для UI он слишком широкий без отдельного решения по взрослым/детским размерам.
