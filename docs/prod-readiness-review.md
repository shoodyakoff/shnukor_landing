## 2. Находки

Severity: **P0** — блокер релиза · **P1** — закрыть до прода · **P2** — желательно/тех-долг.

### 2.1 Безопасность

#### SEC-1 (P0) — открытый токен-прокси `/api/sneakers`
Файл: `src/routes/api/sneakers.ts`

Эндпоинт подставляет приватный `SNEAKERS_API_TOKEN` к каждому запросу и при этом:
- отдаёт `Access-Control-Allow-Origin: *` — **любой** сторонний сайт может вызывать прокси из браузера;
- **не валидирует тело** — `payload` форвардится в апстрим как есть (`JSON.stringify(payload)`);
- **нет rate-limit** и лимита размера тела.

**Риск:** прокси превращается в бесплатный шлюз к платному/квотируемому апстриму под вашим токеном; возможны абуз, исчерпание квоты, скрейпинг каталога, рост счёта.
**Рекомендация:**
1. Убрать wildcard CORS. Фронт ходит на **тот же origin**, поэтому CORS не нужен вовсе — либо отдавать конкретный origin прод-домена. Так же сузить `OPTIONS`.
2. Валидировать тело через **zod** (уже в зависимостях) по контракту из `README_DEPLOY.md`: `size`, `color`, `categories`, `price_from`, `price_to`, `limit` (cap, напр. ≤ 50), `offset`. Лишние поля отбрасывать.
3. Добавить базовый rate-limit (по IP) и лимит размера тела.
4. Опционально: проверять `Origin`/`Referer` и таймаут на `fetch` к апстриму.

#### SEC-2 (P1) — нет заголовков безопасности
Нет CSP, `X-Content-Type-Options`, `Referrer-Policy`, защиты от кликджекинга (`X-Frame-Options` / CSP `frame-ancestors`), HSTS.
**Рекомендация:** добавить заголовки на уровне платформы (`vercel.json` → `headers`) или в Nitro. Минимум: `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `frame-ancestors 'none'`, HSTS. CSP — по возможности (учесть Google Fonts и R2-картинки).

#### SEC-3 (P2) — мёртвый `dangerouslySetInnerHTML`
Файл: `src/components/ui/chart.tsx:73`. Единственное использование, компонент в лендинге не задействован, ключи контролируемы → реальный риск низкий. Уйдёт вместе с прунингом неиспользуемого UI (ARCH-1).

### 2.2 Деплой и инфраструктура

#### DEPLOY-1 (P0) — два противоречащих таргета деплоя
- `.github/workflows/pages.yml` — на push в `main` собирает и публикует **статику** `dist/client` на **GitHub Pages**.
- `README_DEPLOY.md` — прод это **Vercel** (нужен серверный рантайм для прокси), и прямо сказано «Do not deploy as static GitHub Pages».

На GitHub Pages серверной функции `/api/sneakers` нет → подбор всегда уходит в fallback (пустой каталог). Сейчас «прод по умолчанию» — сломанная статика.
**Рекомендация:** выбрать **одну** целевую платформу.
- Если прод = Vercel: удалить `pages.yml` (или переделать в CI: `lint` + `tsc` + `build`, без деплоя), завести прод-домен.
- `wrangler.jsonc` (Cloudflare) при `cloudflare:false` в `vite.config.ts` — рудимент, тоже убрать, чтобы не путать.

#### DEPLOY-2 (P2) — base path и env
`VITE_BASE_PATH` по умолчанию `/shnukor_landing/` (под Pages), на Vercel переключается на `/`. После выбора платформы зафиксировать base явно и проверить, что префикс не протекает в ссылки/ассеты. Также задокументировать обязательные env в одном месте (см. DOC-1).

### 2.3 Код и архитектура

#### ARCH-1 (P2) — большой неиспользуемый UI/зависимости
В `src/components/ui/` ~50 shadcn-компонентов и ~30 `@radix-ui/*` пакетов, лендинг использует малую часть. Бандл tree-shake'ится, но это лишняя поверхность поддержки и обновлений.
**Рекомендация:** удалить неиспользуемые `ui/*` и соответствующие зависимости (можно через `knip`/`depcheck`). Заметно сократит `package.json` и аудит.

Структура `src/components/shnurok/*` (Flow как стейт-машина по `step`, экраны, mapping, api) — логичная и читаемая, замечаний нет.

### 2.4 Тесты и CI

#### QA-1 (P1) — тесты не запускаются
5 файлов `*.test.*`, но в `package.json` **нет тест-раннера** (vitest/jest/@testing-library) и нет скрипта `test`. `tsc --noEmit` даёт 4 ошибки типов — все в тест-файлах (`responsive-layout.test.tsx`, `style-deck-ui.test.tsx`), т.е. тесты разошлись с типами и никто этого не видит.
**Рекомендация:** либо подключить **vitest** (+ `test` скрипт, + `tsc` в CI), либо удалить мёртвые тесты. Половинчатое состояние хуже обоих вариантов.

### 2.5 SEO / мета / контент

#### SEO-1 (P1) — OG/Twitter-картинка ведёт на dev-бакет Lovable
Файл: `src/routes/__root.tsx:51` — `og:image`/`twitter:image` указывают на `https://pub-...r2.dev/...lovable.app...png` (артефакт скаффолда Lovable). Внешний нестабильный URL → битые превью в соцсетях.
**Рекомендация:** положить свою картинку-превью в `public/` и сослаться на абсолютный прод-URL.

#### SEO-2 (P1) — нет favicon / robots.txt / sitemap.xml / manifest
В `public/` их нет, в `__root.tsx` нет `<link rel="icon">`.
**Рекомендация:** добавить favicon (+ apple-touch-icon), `robots.txt`, `sitemap.xml`, по желанию `site.webmanifest`.

#### META-1 (P2) — несогласованный бренд и дублирование мета
Бренд пишется по-разному: `__root.tsx` — «Shnurok», `index.tsx` — «Шнурок»; мета (title/description/OG) задаётся и в root, и в index.
**Рекомендация:** выбрать одно написание бренда; общие мета держать в `__root.tsx`, в роутах переопределять только специфичное.

### 2.6 Мобильная адаптивность
