# Shnurok landing: техническое описание текущей реализации

Актуально на 2026-06-02. Документ описывает фактическое поведение сайта по текущему коду проекта, а не желаемую будущую версию.

Основные источники в коде:

- `src/components/shnurok/Flow.tsx` - порядок экранов и состояние подбора.
- `src/components/shnurok/types.ts` - размеры, цвета, цены, виды спорта.
- `src/components/shnurok/sneakers-mapping.ts` - маппинги карточек, категорий, цветов и сборка API payload.
- `src/components/shnurok/sneakers-api.ts` - клиентский запрос, нормализация товаров и дедупликация.
- `src/routes/api/sneakers.ts` - server-side proxy к API Shnurok.
- `src/components/shnurok/FlowResultScreens.tsx` - поиск, empty-состояние и финальная выдача.
- `src/components/shnurok/ProductCards.tsx` - карточка товара.

## 1. Назначение сайта

Сайт помогает пользователю подобрать кроссовки Shnurok по размеру, цвету, бюджету, задаче и визуальным Tinder-карточкам стиля.

Пользователь проходит последовательный квиз. После последней Tinder-карточки сайт отправляет запрос в API Shnurok через собственный server-side proxy `/api/sneakers`. В финале пользователь получает плоскую сетку товаров с кнопками перехода на карточки товара на сайте Shnurok.

## 2. Общий пользовательский сценарий

Текущая последовательность экранов:

1. Hero.
2. Выбор размера.
3. Выбор цвета.
4. Выбор бюджета.
5. Выбор задачи: `На каждый день` или `Для спорта`.
6. Если выбрано `Для спорта`, дополнительно выбирается конкретный спорт.
7. Tinder-карточки стилей.
8. Экран поиска.
9. Финальная выдача или empty-состояние.

Экран `SummaryScreen` есть в коде, но в текущем `Flow.tsx` не используется в маршруте пользователя.

## 3. Состояние подбора

Выбор пользователя хранится в объекте `Selections`:

```ts
{
  sizes: string[];
  colors: string[];
  price?: string;
  task?: "daily" | "sport";
  sport?: string;
  styleVotes: Record<string, "like" | "dislike">;
}
```

Начальное состояние:

```ts
{
  sizes: [],
  colors: [],
  price: "p3",
  styleVotes: {}
}
```

То есть бюджет по умолчанию уже выбран как `10 000 - 15 000 ₽`, но price chip показывается только начиная с шага цены.

## 5. Цвета

В интерфейсе доступны:

| UI id | Название в UI | Значение для API |
| --- | --- | --- |
| `beige` | Бежевый | `бежевый` |
| `white` | Белый | `белый` |
| `yellow` | Желтый | `желтый`, `золотой` |
| `green` | Зелёный | `зеленый`, `оливковый`, `камуфляжный` |
| `brown` | Коричневый | `коричневый`, `медный` |
| `red` | Красный | `красный`, `бордовый` |
| `orange` | Оранжевый | `оранжевый` |
| `pink` | Розовый | `розовый` |
| `purple` | Фиолетовый | `фиолетовый` |
| `black` | Чёрный | `черный` |
| `grey` | Серый | `серый`, `серебристый` |
| `blue` | Синий | `синий`, `бирюзовый` |
| `any` | Без разницы | цвет в API не отправляется |

Текущее поведение:

- Можно выбрать до 3 конкретных цветов.
- Если выбран `Без разницы`, он сбрасывает все конкретные цвета.
- Если выбран конкретный цвет после `Без разницы`, `Без разницы` убирается.
- Пока не выбран хотя бы один цвет или `Без разницы`, кнопка `Далее` заблокирована.
- В API отправляются русские lowercase-названия цветов из таблицы выше.
- Если выбран `Без разницы`, поле `color` полностью отсутствует в API payload.
- В финальной выдаче выбранный цвет показывается в chips, например `Цвет: Фиолетовый`.

## 6. Бюджет

В интерфейсе доступны диапазоны:

| UI id | Label | API поля |
| --- | --- | --- |
| `p2` | 5 000 - 10 000 ₽ | `price_from: 5000`, `price_to: 10000` |
| `p3` | 10 000 - 15 000 ₽ | `price_from: 10000`, `price_to: 15000` |
| `p4` | 15 000 - 20 000 ₽ | `price_from: 15000`, `price_to: 20000` |
| `p5` | 20 000 - 30 000 ₽ | `price_from: 20000`, `price_to: 30000` |
| `p6` | 30 000 ₽ и выше | `price_from: 30000`, без `price_to` |
| `any` | Без разницы | цена в API не отправляется |

Текущее поведение:

- По умолчанию выбран `p3` (`10 000 - 15 000 ₽`).
- Price chip не показывается на шагах `size` и `color`.
- Начиная с шага `price`, price chip показывается в шапке/результатах.
- На шаге бюджета справа показывается preview-карточка уровня моделей в выбранном диапазоне.
- При `any` поля `price_from` и `price_to` не отправляются.

## 7. Задачи подбора

Пользователь выбирает одну задачу:

| UI label | Internal task | Дальше |
| --- | --- | --- |
| На каждый день | `daily` | сразу Tinder-карточки daily |
| Для спорта | `sport` | выбор конкретного спорта, затем Tinder-карточки спорта |

При смене задачи очищаются предыдущие голоса Tinder-карточек.

## 8. Виды спорта

Для задачи `Для спорта` доступны:

| Спорт | Короткое описание в UI |
| --- | --- |
| Фитнес / зал | стабильная база |
| Бег | легкая амортизация |
| Футбол | низкий цепкий силуэт |
| Трейл / улица | цепкость и защита |
| Баскетбол | поддержка и резкость |
| Теннис | плотная боковая опора |
| Скейтбординг | контроль доски |

Текущее поведение:

- Пока спорт не выбран, кнопка `Далее` заблокирована.
- При выборе спорта очищаются голоса Tinder-карточек.
- В API отправляются категории только выбранного спорта.
- В финальной выдаче chip показывает задачу `Для спорта`; конкретный спорт сейчас в финальном блоке chips не выводится.

## 9. Tinder-карточки и логика выбора категорий

Пользователь проходит все карточки выбранного сценария. Для каждой карточки доступно:

- `Нравится` - голос `like`.
- `Не нравится` - голос `dislike`.
- Также карточку можно свайпнуть.

Текущее правило:

- Если пользователь лайкнул хотя бы одну карточку, в API отправляются только категории лайкнутых карточек.
- Если пользователь не лайкнул ни одной карточки и скипнул все карточки, включается fallback:
  - для `На каждый день` отправляются все 5 daily-карточек;
  - для `Для спорта` отправляются все карточки выбранного спорта.
- All-skip не считается empty-состоянием.
- При all-skip в финальной выдаче показывается текст: `Жаль, что вам не понравился ни один из стилей, вот что мы вам предлагаем`.
- При all-skip подзаголовок: `Мы расширили подбор до всех карточек выбранного сценария и сохранили ваши фильтры.`

## 12. API request

Фронт отправляет POST на локальный proxy:

```txt
/api/sneakers
```

Фактический upstream:

```txt
https://shnurok-shipping.ru/api/sneakers.php
```

Proxy добавляет приватный заголовок авторизации:

```txt
Authorization: Bearer <SNEAKERS_API_TOKEN>
```

Если в `SNEAKERS_API_TOKEN` уже есть префикс `Bearer `, proxy не дублирует его.

Известные поля request:

| Поле | Тип | Когда отправляется |
| --- | --- | --- |
| `size` | `string[]` | если выбран хотя бы один размер |
| `color` | `string[]` | если выбран конкретный цвет; не отправляется при `Без разницы` |
| `categories` | `number[]` | если выбран сценарий/спорт и есть категории |
| `price_from` | `number` | если выбран диапазон с нижней границей |
| `price_to` | `number` | если выбран диапазон с верхней границей |
| `limit` | `number` | всегда `20` |
| `offset` | `number` | всегда `0` |

Пример daily all-skip с размером 42/44, цветом Фиолетовый и бюджетом 10 000 - 15 000 ₽:

```json
{
  "size": ["42", "44"],
  "categories": [349, 731, 730, 569, 576],
  "color": ["фиолетовый"],
  "price_from": 10000,
  "price_to": 15000,
  "limit": 20,
  "offset": 0
}
```

Если выбран цвет `Без разницы`, тот же payload будет без поля `color`.

Если выбран бюджет `Без разницы`, payload будет без `price_from` и `price_to`.

Если выбран бюджет `30 000 ₽ и выше`, payload будет с `price_from: 30000` и без `price_to`.

## 13. API response

Текущий ожидаемый ответ API:

```json
{
  "success": true,
  "count": 379,
  "limit": 20,
  "offset": 0,
  "items": [
    {
      "id": 4073,
      "offer_id": 161829,
      "name": "Adidas Ozweego Core Black",
      "price": 10490,
      "currency": "RUB",
      "size": "42",
      "image": "https://shnurok-shipping.ru/upload/...",
      "detail_url": "https://shnurok-shipping.ru/catalog/..."
    }
  ]
}
```

Клиент умеет извлекать массив товаров из полей:

- корневой массив;
- `items`;
- `data`;
- `products`;
- `sneakers`;
- `result`;
- `results`.

Известные поля товара:

| Поле API | Использование в UI |
| --- | --- |
| `id` | fallback id товара |
| `offer_id` | основной id товара |
| `name` | название товара |
| `price` | цена, форматируется в `ru-RU` с `₽` |
| `currency` | сейчас напрямую в UI не используется |
| `size` | размер в карточке товара |
| `image` | изображение товара |
| `detail_url` | ссылка CTA `Перейти в магазин` |

Также нормализатор поддерживает альтернативные поля:

- бренд: `brand`, `manufacturer`, `vendor`, `marka`;
- название: `name`, `title`, `model`, `product_name`;
- изображение: `img`, `image`, `photo`, `picture`, `thumbnail`, `image_url`, первый элемент `images` или `photos`;
- ссылка: `detail_url`, `url`, `link`, `href`, `product_url`;
- цена: `price`, `cost`, `amount`;
- описание: `why`, `description`, `short_description`;
- fit/score: `fit`, `match`, `score`.

Важно: текущая финальная карточка не показывает `why`, `fit`, `match` или `% подходит`.

## 14. Нормализация и дедупликация товаров

После ответа API клиент:

1. Извлекает массив товаров.
2. Нормализует каждую запись в `ProductItem`.
3. Убирает товары без названия.
4. Дедуплицирует по нормализованному ключу:
   - сначала `url`;
   - если ссылки нет, `name`;
   - если имени нет, `id`.
5. Нормализует относительные ссылки относительно `https://shnurok-shipping.ru`.
6. Если изображения нет, использует локальные fallback-картинки.

Требование текущей реализации: в выдаче не должно быть дублей одного товара после нормализации URL/имени/id.

## 15. Распределение выдачи

Фронт не распределяет товары между категориями и цветами самостоятельно.

Текущее поведение:

- Фронт всегда отправляет `limit: 20` и `offset: 0`.
- Если выбрано несколько категорий и/или цветов, вся пропорциональная логика находится на стороне upstream API.
- Фронт отображает то количество товаров, которое API вернул после фильтров и дедупликации.
- Если API вернул меньше 20 товаров, финальная выдача показывает все доступные товары.

Пример из текущей проверки:

- payload без цвета для daily all-skip, размеров 42/44 и бюджета 10 000 - 15 000 ₽ возвращает `count: 379`, `items.length: 20`;
- тот же payload с `color: ["фиолетовый"]` возвращает `count: 4`, `items.length: 4`.

## 16. Финальная выдача

Финальная выдача сейчас плоская, без группировок.

Текущее поведение:

- Нет групп `Точное попадание` и `Еще варианты`.
- Нет группировки по Tinder-категориям, потому что API не возвращает `matched_categories`, `category_id`, `primary_matched_category` или аналогичные поля.
- Заголовок обычной выдачи: `Нашли N моделей под твой запрос`.
- Заголовок all-skip выдачи: `Жаль, что вам не понравился ни один из стилей, вот что мы вам предлагаем`.
- Подзаголовок обычной выдачи: `Учли доступные фильтры запроса.`
- Подзаголовок all-skip выдачи: `Мы расширили подбор до всех карточек выбранного сценария и сохранили ваши фильтры.`
- В chips финальной выдачи показываются:
  - размер;
  - цвет;
  - цена;
  - задача.
- Конкретный спорт в финальном блоке chips сейчас не показывается.

Карточка товара показывает только:

- изображение;
- бренд, если он есть в нормализованном товаре;
- название;
- цену;
- размер;
- CTA.

CTA:

- если есть `url`, показывается ссылка `Перейти в магазин`, открывается в новой вкладке;
- если ссылки нет, показывается disabled-like текст `Ссылка скоро появится`.

Карточка товара не показывает:

- `% подходит`;
- блок `Почему подходит`;
- hashtags;
- объяснение матчинга;
- группировку результата.

## 17. Empty-состояние

Empty-состояние показывается только если:

- API успешно ответил;
- `apiResult.source === "api"`;
- API вернул `0` товаров;
- это не all-skip сценарий.

Текст empty-состояния:

```txt
По вашим параметрам ничего не найдено
Давайте попробуем еще раз или перейдите на сайт Шнурок https://shnurok-shipping.ru/
```

На empty-экране есть действия:

- `Поднять цену`;
- `Убрать цвет`;
- `Смягчить стиль`;
- `Изменить задачу`;
- `Другой размер`;
- `Показать ближайшие варианты`.

В текущей реализации эти действия не меняют фильтры автоматически. Они переводят пользователя к результатам/ближайшим вариантам через текущий обработчик.

All-skip не считается empty-состоянием. При all-skip пользователь всегда переходит в results-экран.

## 18. API failure fallback

Если клиентский запрос к `/api/sneakers` падает или ответ невозможно обработать:

- `fetchSneakers` возвращает `{ items: [], source: "fallback" }`;
- empty-состояние не показывается, потому что source не `api`;
- финальный экран при пустом массиве `items` использует локальный `CATALOG_ITEMS`.

Это временная защитная логика, чтобы пользователь не упирался в ошибку API.

## 19. Товарные fallback-данные

Локальный fallback-каталог используется:

- на loading/search экране как preview;
- в финальной выдаче, если `Results` получил пустой массив товаров.

Fallback-товары находятся в `src/components/shnurok/data.ts`.

Локально:

## 21. Проверяемые требования в автотестах

В текущих тестах покрыто:

- если лайкнута хотя бы одна daily-карточка, в payload уходят только лайкнутые категории;
- daily all-skip отправляет все 5 daily-категорий;
- sport-сценарий отправляет категории выбранного спорта;
- sport all-skip отправляет все категории выбранного спорта;
- `Без разницы` по цене убирает price-фильтры;
- `30 000 ₽ и выше` отправляет только `price_from`;
- daily-карточки соответствуют `[349, 731, 730, 569, 576]`;
- сценарий `Универсальные для спорта` отсутствует;
- карточка `Дутые` отсутствует;
- price chip скрыт до шага цены;
- manual API payload читается обратно в `Selections`;
- товары дедуплицируются;
- ссылки товара нормализуются;
- proxy добавляет `Bearer`, если токен хранится без префикса;
- финальная карточка товара не показывает `% подходит` и `Почему подходит`;
- финальная выдача показывает все товары плоским списком без `Точное попадание` и `Еще варианты`;
- финальная выдача показывает выбранный цвет.

## 22. Текущие ограничения

1. API не возвращает matched category по каждому товару. Поэтому нельзя честно группировать результаты по Tinder-карточкам.
2. API не возвращает объяснение матчинга. Поэтому UI не показывает `Почему подходит`.
3. Фронт не балансирует выдачу по категориям/цветам сам; баланс ожидается от upstream API.
4. Финальный экран не показывает конкретный выбранный спорт в chips, только задачу `Для спорта`.
5. Empty actions визуально есть, но не изменяют фильтры автоматически.
6. `SummaryScreen` есть в коде, но не подключен в основной flow.
7. При API failure пользователь получает локальный fallback-каталог, а не техническую ошибку.
8. Большие PNG-ассеты в `public` увеличивают размер production assets.
9. `npm audit --omit=dev` на 2026-06-02 показывает moderate vulnerabilities в dependency tree; перед строгим production hardening желательно обновить зависимости и перепроверить сборку.

## 23. Требования к будущей группировке по Tinder-карточкам

Чтобы можно было группировать финальную выдачу по выбранным Tinder-карточкам, API должен возвращать для каждого товара одно или несколько полей:

```json
{
  "matched_categories": [349, 731],
  "primary_matched_category": 349
}
```

Где:

- `matched_categories` - пересечение категорий товара с категориями из запроса;
- `primary_matched_category` - категория, в которую backend положил товар для пропорциональной выдачи.

Без этих полей фронт не должен группировать товары по Tinder-карточкам, потому что это будет догадкой по URL или названию.
